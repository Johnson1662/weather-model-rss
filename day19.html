今日主题：系统集成与自动化 — 从零搭建"压缩-量化-部署"实验管线

前 18 天我们学了很多"零件"：剪枝、量化、低秩分解、ONNX 导出、GPU 调优、国产卡适配……现在到了关键时刻——如何把这些零件拼成一条自动化的流水线？

这就好比厨房里你学会了切菜、炒菜、调味、摆盘，但每天点餐的人来了，你还得手忙脚乱地一样一样做。而比赛的时间窗口只有几天，你需要的是一条完整的"中央厨房"流水线——输入原始模型，一键输出压缩好、量化好、可在国产卡上运行的最终模型。

今天的内容就是教你怎么搭建这条流水线。

一、从大赛的工作流程说起

先导杯的推理部署优化任务，本质上是一个实验迭代循环：

┌─────────────────────────────────────────────────────┐
│           比赛工作循环（一次迭代）                      │
├─────────────────────────────────────────────────────┤
│                                                       │
│  原始模型（Pangu-Weather）                             │
│       │                                               │
│       ▼                                               │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐        │
│  │  剪枝     │ →  │  量化     │ →  │  ONNX    │        │
│  │ (结构化)  │    │ (PTQ/SQ) │    │  导出    │        │
│  └──────────┘    └──────────┘    └──────────┘        │
│       │                                               │
│       ▼                                               │
│  ┌────────────────────┐                               │
│  │  国产卡推理验证     │                               │
│  │  (精度 + 速度)     │                               │
│  └────────────────────┘                               │
│       │                                               │
│       ▼                                               │
│  ┌────────────────────┐                               │
│  │  结果记录 & 决策    │  ← 今天讲的核心               │
│  │  (继续优化/提交)   │                               │
│  └────────────────────┘                               │
│                                                       │
└─────────────────────────────────────────────────────┘

每次从"原始模型"到"国产卡上跑出结果"是一个完整的实验。你需要跑很多次这种实验，调整不同的剪枝率、不同的量化策略……然后比较哪一次最好。

不自动化会怎样？

  • 手动修改参数 → 忘记改了什么 → 两个实验结果没法比
  • 每次跑完靠记忆写结果 → 写到第三周全忘了
  • 队友问"昨天那个剪枝 30% 的实验参数是什么" → 你翻半天没找到
  • 提交截止前才发现错了 → 没时间重跑

所以你需要一个实验管理系统。任务优先级的排序：

  1. 能记录：每个实验的输入（参数）+ 输出（指标）都能保存
  2. 能查找：任何时候都能找到"哪个实验最好"
  3. 能复现：同样的参数输入，随时能轻松跑出同样结果
  4. 能对比：一眼看出不同策略之间的差异

二、实验跟踪（Experiment Tracking）的原理

想象你在做蛋糕。从头到尾你做了 10 个蛋糕：

  • 第 1 次：面粉 200g，糖 100g → 口感偏干
  • 第 2 次：面粉 180g，糖 100g，加一个鸡蛋 → 口感松软
  • 第 3 次：面粉 200g，糖 120g → 甜但干

用脑子记这些，到了第 5 个就乱了。你需要一张表：

实验编号面粉(g)糖(g)鸡蛋口感评分
120010006/10
218010018/10
320012005/10

实验跟踪就是为每个模型实验建这样一张卡片，记录所有相关信息。

在比赛语境下，一张实验卡片需要记录什么？

类别具体字段为什么重要
基本信息实验名称、时间戳、运行人方便查找和区分
模型信息模型名、参数量、显存占用判断压缩效果
剪枝参数剪枝率、剪枝目标（head/channel）、评分方法复现剪枝实验
量化参数量化精度、是否 SmoothQuant、校准集复现量化实验
推理性能延迟(ms)、吞吐量(帧/秒)、显存峰值比赛评分核心
预报精度RMSE、ACC（按变量分别）比赛评分核心
部署信息推理引擎、国产卡类型、ONNX opset可复现性

比赛中的意义：每次跑完实验，把上面这张表的字段填好。坚持填 3 天，你就会发现自己对哪个策略有效、哪个无效已经有了清晰判断。没有实验跟踪，你就是在"蒙眼做优化"。

三、实验跟踪的三种实现方式

方法 1：手动填表（最简单，适合前 5 个实验）

打开一个 Excel/Google Sheets，列头就是你上面那张表的字段。每次跑完实验，手动填一行。

| 实验名称 | 剪枝率 | 量化精度 | RMSE(Z500) | ACC(Z500) | 延迟(ms) | 参数量 |
|---------|-------|---------|-----------|----------|--------|-------|
| baseline| 0%    | FP32    | 0.0031    | 0.987    | 850    | 64M   |
| exp01   | 10%   | FP32    | 0.0035    | 0.983    | 720    | 58M   |
| exp02   | 10%   | INT8    | 0.0042    | 0.978    | 380    | 58M   |
| exp03   | 20%   | INT8    | 0.0051    | 0.970    | 310    | 51M   |

优点：零成本、随时可用。
缺点：忘记填就没了、不好对比、多人协作麻烦。

方法 2：代码自动记录（推荐，适合比赛）

写一个简单的 Python 脚本，每次跑实验时自动把参数和结果写进一个 JSON 文件。

{
  "experiments": [
    {
      "id": "exp_001",
      "timestamp": "2026-07-03 14:30:00",
      "config": {
        "pruning_ratio": 0.1,
        "pruning_target": "head",
        "importance_score": "taylor",
        "quantization": {"type": "INT8", "calibration": "500_samples"},
        "smoothquant": true
      },
      "metrics": {
        "params": 58000000,
        "memory_mb": 1840,
        "latency_ms": 720,
        "throughput_fps": 1.39,
        "rmse_z500": 0.0035,
        "acc_z500": 0.983,
        "rmse_t850": 0.0028
      },
      "status": "completed",
      "onnx_file": "models/exp_001.onnx"
    }
  ]
}

每次实验结束，用 Python 写一个 append_experiment() 函数把新行追加进去。配合一个小脚本 compare.py 就能自动生成对比表。

方法 3：专业跟踪工具（MLflow / Weights & Biases）

像 MLflow（开源）这样的工具为你提供完整的实验管理平台：

  • 每次运行自动记录参数、指标、模型文件
  • 你可以在 Web 界面中对比不同实验的曲线
  • 支持团队共享和一个"模型注册表"（记下哪个模型最好）

比赛中的意义：推荐从方法 2（JSON 自动记录）开始，不依赖外部服务、不怕断网、格式透明。如果团队大、实验多，再升级到方法 3。

四、设计"压缩-量化-部署"流水线

流水线的本质是：把人工操作步骤变成脚本命令。一条命令就能完成从"原始模型"到"国产卡上跑指标"的全流程。

一条典型流水线的阶段划分：

阶段1：模型准备
  └─ load Pangu-Weather 预训练权重
  └─ 剪枝（指定剪枝率和目标）
       → 输出: pruned_model.pth

阶段2：模型压缩
  └─ 量化（指定精度和校准集）
       → 输出: quantized_model.onnx
  └─ onnxsim 简化
       → 输出: simplified_model.onnx

阶段3：国产卡部署验证
  └─ ONNX Runtime（国产卡 EP）
  └─ 推理测试（指定测试集）
       → 输出: latency, memory, pred_output.npy

阶段4：精度评估
  └─ 与真值对比，计算 RMSE / ACC
       → 输出: metrics.json

阶段5：结果记录
  └─ 拼接所有指标和参数
  └─ 追加到实验跟踪文件（JSON/CSV）
       → 输出: experiments.json

把这 5 个阶段写成 5 个脚本（01_prune.py, 02_quantize.py, 03_deploy.py, 04_evaluate.py, 05_log.py），每个脚本独立可运行。

然后用一个主控脚本串联它们：

# run_pipeline.sh — 一条命令跑完整流程
python 01_prune.py   --ratio 0.2  --target head    \
  --input model.pth --output models/pruned_20.pth

python 02_quantize.py --precision INT8             \
  --input models/pruned_20.pth                     \
  --calib data/calib_500.npz                       \
  --output models/exp_001.onnx

python 03_deploy.py --engine onnxruntime           \
  --model models/exp_001.onnx                      \
  --backend domestic --output outputs/exp_001/

python 04_evaluate.py --pred outputs/exp_001/      \
  --groundtruth data/truth.npz                     \
  --output metrics/exp_001.json

python 05_log.py --config exp_001.json             \
  --metrics metrics/exp_001.json                   \
  --dump experiments.json

关键设计原则：

  1. 单一入口：修改参数只需改一行（比如把 --ratio 0.2 改成 --ratio 0.3），而不是在代码里搜 search 替换
  2. 模块可替换：如果某天你换了一个量化方法（从 PTQ 换成 SmoothQuant），只需改 02_quantize.py，其他脚本不动
  3. 可恢复：如果阶段 3 挂了，修好后重新执行 03_deploy.py 就行，不用从头跑
  4. 隐式版本：每个实验的输出放在 outputs/exp_XXX/ 文件夹下，不会覆盖之前的实验

比赛中的意义：同一个团队里，有人负责剪枝、有人负责量化。流水线让每个人只管自己的模块，最后在run_pipeline.sh里拼接。这对团队分工至关重要。

五、版本管理：确保每次实验都可复现

版本管理的目标：半年后翻出这个实验，我还能一模一样地跑出来。

需要版本控制的内容：

要跟踪什么用什么管理为什么要管
源代码（脚本、配置文件）Git改了什么代码一清二楚
实验配置（剪枝率、量化精度）JSON/YAML 配置文件 + 实验跟踪表复现实验
模型权重/ONNX 文件Git LFS 或共享文件夹大文件的版本管理
实验记录（指标、图表）JSON/CSV 文件比较不同策略

Git 在比赛中的最小用法：

# 每次跑实验前，确保代码是干净的
git status

# 代码有改动？先提交
git add .
git commit -m "feat: 添加 SmoothQuant 支持"

# 跑实验
bash run_pipeline.sh --ratio 0.2

# 跑完后，提交实验记录
git add experiments.json outputs/exp_001/
git commit -m "exp_001: 剪枝20% + INT8 量化"

这样每个 Git commit 都对应一个实验。你想找回任意一个实验的原始状态：

git checkout <那个实验的commit>
# 跑一遍，一模一样的结果

比赛中的意义：提交截止前发现某个结果有问题，你能通过 Git 快速定位"是哪次改动导致了精度下降"，而不是抓瞎从头排查。

六、一键导出的打包流程

比赛提交的最终产物是一个可以被推理引擎直接加载的模型文件（通常是 ONNX）和一个推理脚本。你需要一个打包流程来确保：

  • 导出的 ONNX 文件能在目标国产卡上跑通
  • 推理脚本的参数、路径都是正确的
  • 提交的模型就是"你验证过的最好模型"

标准打包流程：

Step 1: 从实验跟踪表中选出"最佳模型"
  │  条件：RMSE ≤ baseline × 1.05 且 延迟 ≤ baseline × 0.5
  │  结果：确定 exp_012 是最优组合
  ▼
Step 2: 导出提交版 ONNX
  │  固定输入 shape（静态 shape 最优）
  │  检查 ONNX Opset 与国产卡一致
  ▼
Step 3: ONNX 兼容性验证
  │  onnxsim 简化
  │  onnxruntime 下跑一次推理，对比输出
  ▼
Step 4: 打包提交文件
  │  submit/
  │  ├── model.onnx        ← 优化后的 ONNX 模型
  │  ├── inference.py       ← 推理入口脚本
  │  ├── requirements.txt   ← 依赖清单
  │  └── README.md          ← 使用说明
  ▼
Step 5: 国产卡上做最终验证
  │  在国产卡上跑 inference.py
  │  检查输出和精度是否一致
  ▼
提交 ✅

比赛中的意义：很多队伍最后时刻才发现模型导出的 ONNX 在国产卡上跑不了——因为某个算子不支持、因为动态 shape 没固定、因为 opset 版本不一致。"一键打包"流程能在提交前自动检查这些问题。

八、常用命令行工具速查

任务命令说明
查看所有实验ls experiments/*.json列出所有实验记录
对比所有实验的 RMSEjq '.accuracy_metrics.z500.rmse' experiments/*.json一行命令出对比
对比所有实验的延迟jq '.perf_metrics.latency_ms.mean' experiments/*.json一行命令出对比

获取 Git 当前状态git log --oneline -5看最近 5 次提交

比赛中的意义：比赛最后几天时间紧张。你能用一行命令批量跑 5 个实验并自动记录结果，就比手动跑 5 个实验的队友多出 3 个小时来做分析和答辩准备。

九、常见误区

  1. "实验跟踪太麻烦，我用脑子记就行"——第 1-3 个实验还行，第 10 个之后你连哪个剪枝率配哪个量化都分不清。
  2. "Python 脚本在 Jupyter Notebook 里写死了参数，不用配置文件"——改参数要在代码里搜索替换，容易漏改。永远把参数写在命令行或 YAML 文件里。
  3. "只用 Git 管代码就够了"——Git 管的是代码版本，你还需要单独管理实验配置和实验结果的对应关系。
  4. "流水线写好了就要一次跑通全流程"——流水线是为了方便迭代。每次改动一个参数就全跑可能太慢，可以只跑受影响的部分。
  5. "比赛最后再统一做实验记录"——到时你会发现自己根本记不清近 20 天的实验细节。每天跑完立刻记录。
  6. "实验文件名带 final 就能分清版本"——只会出现 model_final.onnx, model_final_v2.onnx, model_final_v3_really.onnx。用编号才是正道。
